# Documentation Standards

1. Update docs when behavior or contracts change.
2. Keep examples executable and current.
3. Keep current live standards in `docs/standards/`.
4. Treat docs under `docs/archive/` as archived read-only reference.
5. Prefer positive responsibility statements; avoid "what this service does not do" sections unless needed to prevent a boundary mistake.
6. Avoid historical rename notes in active specs; use current canonical names only.
7. When you change something runnable, include copy-paste repo commands in your response so the change can be verified locally.
